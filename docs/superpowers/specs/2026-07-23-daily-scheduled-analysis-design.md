# 每日 22:30 大盘与股票列表分析设计

日期：2026-07-23
状态：待书面确认

## 目标

每天北京时间 22:30 自动运行 GitHub Actions，先完成大盘分析并推送，再完成股票列表分析并推送。周末和非交易日仍触发，程序使用最近一个完整交易日的数据。

## 调度与时区

GitHub Actions cron 使用 UTC，因此设为 `30 14 * * *`，即每天北京时间 22:30，不再限制星期一至星期五。

## 执行结构

保留 `.github/workflows/00-daily-analysis.yml` 和单个 `analyze` job。对于 `schedule` 事件，在同一 runner 内按顺序执行：

1. `python main.py --market-review --force-run`；
2. 通过现有 PushPlus 流程推送大盘报告；
3. `python main.py --no-market-review --force-run`；
4. 通过现有 PushPlus 流程推送股票列表报告；
5. 汇总两次命令的退出状态。

顺序执行可以控制模型请求量，并保证微信消息按“大盘在前、个股在后”的顺序到达。

## 失败隔离

两次命令分别记录退出状态。大盘分析失败时仍继续股票列表分析；股票列表分析失败也不影响已经完成的大盘推送。两项都尝试后，只要任意一项失败，job 最终标记失败。

该分支只用于 `schedule` 事件。手动运行继续保留现有 `full`、`market-only`、`stocks-only` 和 `force_run` 选项及其原有行为。

## 超时与资源

将默认 `timeout-minutes` 从 30 提高到 60，并继续允许仓库变量 `ANALYSIS_TIMEOUT_MINUTES` 覆盖默认值。复用单个 runner，不增加并行 job。

## 微信推送

大盘分析和股票列表分析分别调用现有通知流程，形成两组独立推送。个股报告较长时，现有代码可能把它拆成多条微信消息，本次不修改拆分算法。

已知现象：此前 16 只股票测试中，个股报告拆为 6 批，前 5 批成功，第 6 批返回服务端验证错误。本次范围是每日调度和两类分析独立执行；该偶发推送问题作为后续独立事项处理。

## 配置边界

本次只修改工作流调度和 schedule 执行分支，不修改 GLM Coding Plan 模型、API 地址或密钥，不修改 PushPlus token、股票列表、报告生成代码和手动触发参数。

## 验收标准

1. cron 为 `30 14 * * *`，每天触发；
2. schedule 事件依次执行大盘和股票列表分析，二者都带 `--force-run`；
3. 第一项失败时第二项仍执行；
4. 任一项失败时，两项都尝试后 job 标记失败；
5. 默认超时为 60 分钟；
6. 手动触发行为保持不变；
7. 不新增或暴露任何密钥。

## 验证方式

修改后读取 YAML 核对 cron、超时和分支逻辑，检查 YAML 语法，再手动触发一次完整分析，核对两类报告和 PushPlus 推送，并在日志中确认失败隔离逻辑不会提前中止第二项。
